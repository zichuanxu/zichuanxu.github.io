这个项目把经典的“黑马点评”业务迁移到 Go 技术栈：HTTP 层使用 Gin，数据访问使用 GORM 与 MySQL，Redis 负责会话、缓存和原子预检，Kafka 用于探索异步订单链路。真正有价值的部分不是 API 数量，而是同一条业务链路如何随着并发要求逐步改变。

> 本文是对公开项目 [amemiya02/hmdp-go](https://github.com/amemiya02/hmdp-go) 的架构记录。原始业务来自黑马点评课程；这里聚焦 Go 实现中的设计取舍，不把示例工程包装成生产系统。

## 项目范围 {#scope}

项目覆盖四组相互关联的能力：

- Token 登录与 Redis 会话续期；
- 商户查询中的缓存穿透、雪崩和热点 Key 重建；
- 优惠券秒杀中的库存预检、一人一单与异步落库；
- 点赞、共同关注、附近商铺、签到和 UV 统计。

核心请求链路可以压缩成下面这条路径：

## 两层中间件拆开会话刷新与鉴权 {#session}

多实例部署后，进程内 Session 无法让不同节点共享登录状态。项目改为由客户端携带 Token，服务端用 Redis Hash 保存精简后的用户信息，并为会话设置 TTL。

关键不是“把 Session 换成 Redis”，而是把两个职责拆开：

1. **刷新中间件**挂在全局路由上。请求带有有效 Token 时，它读取用户信息、写入 `gin.Context`，并刷新 Redis TTL；没有 Token 时直接放行。
2. **鉴权中间件**只挂在受保护的路由组上。它不再访问 Redis，只检查前一层是否已经把用户写入上下文，失败时返回 `401`。

如果只在下单等受保护接口刷新 TTL，用户长时间浏览公开页面后仍会被动退出。全局刷新解决的是活跃度，鉴权中间件解决的是访问控制；拆开后，每一层都只有一个判断标准。

会话中只保存 `id`、昵称和头像等必要字段，而不是整行用户记录。这样既减少 Redis 占用，也避免把不需要的账户字段带入后续请求上下文。

## 缓存治理不是一个统一答案 {#cache}

商户详情采用 Cache Aside：先读缓存，未命中时回源数据库并写回。这个基础路径需要针对不同失效方式补上不同策略。

| 问题 | 触发方式 | 项目中的处理 |
| --- | --- | --- |
| 缓存穿透 | 反复查询不存在的 ID | 缓存短 TTL 的空值，避免每次访问数据库 |
| 缓存雪崩 | 大量 Key 同时过期 | 在基础 TTL 上增加随机偏移，打散失效时间 |
| 热点 Key 击穿 | 高并发 Key 突然失效 | 使用互斥重建，或返回旧值并异步重建 |

互斥重建更接近一致性优先：只有获得 Redis 锁的请求查询数据库并回填，其他请求短暂等待后重试。它的代价是热点失效时会增加尾延迟。

逻辑过期更接近可用性优先：缓存值同时记录业务过期时间；过期后，请求仍可读取旧值，只有抢到锁的协程在后台重建缓存。当前实现还会设置一个更长的物理 TTL，因此它不是永久缓存。这个细节很重要：逻辑时间决定何时刷新，物理 TTL 仍负责最终回收。

写路径采用“先更新数据库，再删除缓存”。这不能凭空提供强一致性，但比同时维护两份值更简单；如果业务要求更严格，还需要结合版本号、消息补偿或事务化方案，而不是只靠延迟双删。

## 秒杀链路的三次拆分 {#seckill}

秒杀场景同时要求库存不能为负、同一用户不能重复下单，并且入口不能被数据库写入速度拖垮。项目用三个版本探索这条链路。

### V1：同步事务与用户级锁

请求在主链路中完成资格检查、扣减库存和创建订单。单机版本用 `sync.Map` 保存用户级互斥锁，不同用户可以并发，同一用户串行执行；多实例后则改用 Redis 锁。

这个版本路径直观，但数据库 I/O 与锁等待都位于请求周期内，吞吐会直接受最慢步骤限制。

### V2：Redis 预检与内存 Channel

Lua 脚本在 Redis 内原子完成库存检查、一人一单检查和预扣减，然后把订单交给 Channel。入口更短，但内存队列无法跨实例，也会在进程退出时丢失尚未消费的数据。

当前仓库保留了 Channel executor 的实现，但没有形成完整的应用级消费者接线，因此它应当被看作一个中间实验，而不是已经完成的异步订单方案。

### V3：Redis 预检与 Kafka

Kafka 替换进程内队列后，订单可以由独立消费者异步落库，并具备跨实例消费的基础。数据库仍需使用条件更新和唯一约束兜底，因为消息重试可能产生重复消费。

<div class="article-flow" role="list" aria-label="秒杀订单链路">
  <span role="listitem">Redis + Lua 预检</span>
  <i aria-hidden="true">→</i>
  <span role="listitem">Kafka</span>
  <i aria-hidden="true">→</i>
  <span role="listitem">消费者事务</span>
  <i aria-hidden="true">→</i>
  <span role="listitem">MySQL 约束</span>
</div>

这里的重点不是“用了 Kafka”，而是把快速拒绝、流量缓冲和最终写入拆成不同阶段，并让每一层都保留自己的正确性约束。

## Redis 数据结构对应具体查询 {#structures}

项目后半部分没有继续增加通用缓存，而是按查询语义选择数据结构。

| 业务 | Redis 结构 | 选择原因 |
| --- | --- | --- |
| 点赞列表 | ZSet | 用户 ID 去重，并按点赞时间排序 |
| 共同关注 | Set | 使用集合交集直接得到共同用户 |
| 附近商铺 | GEO | 按经纬度做半径查询和距离排序 |
| 月度签到 | BitMap | 用一个 bit 表示一天，便于统计连续签到 |
| UV 估算 | HyperLogLog | 以固定量级内存估算大量独立访客 |

这些结构减少了应用层扫描和排序，但不会替代主数据库。Redis 保存的是为特定查询准备的索引或近似统计，MySQL 仍然承担业务记录的持久化。

## 当前实现的边界 {#limits}

这次重构建立了从同步处理到异步链路的完整思考路径，也留下了几个需要继续收敛的地方：

- 分布式锁的续期与释放必须操作完全相同的 Key；当前 Redisson 风格实现仍需要统一前缀并补充故障测试。
- Channel 版本需要真正启动消费者、处理关闭流程，并定义队列满时的行为。
- Kafka 消费需要明确幂等键、重试和死信处理，而不只是把写入搬到后台。
- 性能数字只有在固定数据量、并发模型和硬件环境下才有意义；本文不把本地测试直接外推为生产容量。

对这个项目而言，下一步最有价值的工作不是再增加一个中间件，而是让现有链路具备可重复的并发测试、明确的失败语义和可观测指标。
